Status of DAnCE

Information for the Status of DAnCE

The purpose of this page is to illustrate what features in the OMG's DnC Spec are implemented/supported and what are not.

Things we have supported and things we have NOT supported yet:

  1. XML Handler
  • XML handlers are capable of parsing all elements that are defined in the XML schema. To be more precise, given a XML descriptor confirming to the XML schema specified by the D&C specification, DAnCE XML handlers can parse all of those elements and store the information in the IDL data structure also specified by the D&C specification.

  • The only restriction is in the elements of the type Deployment::Property. These elements are complex types and can have sub elements. The corresponding IDL data structure, which will capture the values stored in the XML descriptors, are of the type "Any". DAnCE XML handlers, right now, have the capability to store only the basic data types in these "Any"s.

  • DAnCE run-time currently reads the information stored in the "DeploymentPlan" data structure to perform the deployment of components. This capability is currently restricted to reading only the following tags (See the descriptors contained in $CIAO_ROOT/DAnCE/tests/Config_Handlers/HUDisplay directory for details):

    (1) MonolithicDeploymentDescription: Specifies the monolithic implementation description of a component (interface), which is part of an assembly, which in turn is part of a package.

    • User cannot specify the deployRequirement for this monolith. This attribute specifies the target or collocation requirements that this implementation could have. Presently, DAnCE doesn't handle this.
    • User cannot specify any Property (execParameter) for the monolith. This attribute specifies the run-time attributes of this component. Currently, DAnCE run-time doesn't handle the attributes for the components.

    (2) ArtifactDeploymentDescription: Specifies the artifacts used by the monolithic implementation of a component(interface). Common examples are entryPoints of a component and common libraries like ACE+TAO+CIAO which are used by the components.

    • User cannot specify the deployRequirement, deployedResource and execParameter for this artifact presently. To be precise, an user cannot specify the target resource on which this artifact will run, cannot specify the target resources on which this artifact will depend on and cannot specify the run-time attributes this artifact will depend on.

    (3) InstanceDeploymentDescription: Specifies which instance of a component runs on which node, which is defined in the DomainDescriptor.

    • DAnCE presently has the capability only to specify the instance name and the node on which the instance will run.

    (4) PlanConnectionDescription: Specifies the connection between the components, which are part of the application.

    • DAnCE currently supports connections between internalEndpoints only. ComponentExternalPortEndpoint and ExternalReferenceEndpoints are not supported.

    (5) PlanPropertyMapping: Maps the application's to-be-exposed properties onto the properties of the components, part of the application.

    • DAnCE does not support this.

    (6) ImplementationDependencies: Implementations of certain components need to be run on the target environments, before the application deployment, specified by this deployment plan, can be made.

    • DAnCE does not support this.

    (7) Property: DeploymentPlan specific properties.

    • DAnCE does not support this.

    (8) All the "elements" defined in the XML schema can have attributes. However, our XML handlers can handle the following attributes only: xmi:idref and xmi:id. We also don't support the xmi:Extension elements.

  1. Deployment Tools and Interfaces

  • RepositoryManager interface: Only "installPackage" and "findPackageByName" operations are implemented. This interface is implemented as an executable.
  • TargetManager interface: Has not been implemented yet. Implementation of this feature is a work in progress.
  • ExecutionManager is implemented completely as an executable.

  • DomainApplicationManager and DomainApplication interfaces are combined together and is called "DomainApplicationManager". As a result, all the methods, except getApplications (), in the DomainApplicationManager are implemented. Currently, the DomainApplicationManager is implemented as a library and used by ExecutionManager executable.
  • NodeManager is implemented completely as an executable.
  • NodeApplicationManager is implemented completely as a library and used by NodeManager executable.
  • NodeApplication is implemented completely as an executable.
  • Executor Actor defined in the OMG's DnC Spec is implemented as part of the RepositoryManager executable.
