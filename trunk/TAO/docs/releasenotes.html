ORB & ORB Core

Current status:

  • Provides dynamically linked wholesale strategy assignment using the ACE Service Configurator framework.
  • Supports several concurrency models out of the box: Reactive, Thread-per-connection, and ORB-per-thread. Others can be easily crafted.
  • The ORB-per-thread model is not well integrated with the other concurrency models; it must be selected at compile time rather than run time.

Future work:

  • Add Orbix-like bind calls.
  • Better integrate ORB-per-thread.
  • Clean up the internals and analyze for performance bottlenecks using Quantify.
  • Integrate with POA work.
  • Add support for resolve_initial_references.
  • Assist in porting higher level services (e.g., Event Channel) to TAO.

IDL Compiler

Current status

  • Generates stubs and skeletons which use the interpretive IIOP marshaling engine.
  • Works with most IDL constructs. Although code is generated for everything, support for unions, sequences whose element types are anonymous sequences or strings, arrays of strings, and other very complicated types may result in compilation errors.

Future work:

  • Improve IDL compiler
  • Fix bugs in the SunSoft IDL front-end uncovered by us.
  • Use Flick (from the University of Utah) to generate compiled stubs.
  • Goal is to measure the code size of the interpretive stubs generated by TAO IDL compiler vs code size of compiled stubs. Then compare the performance of each. We want to prove the thesis that TAO IDL compiler generated interpretive stubs have a small code size, yet are comparable in performance (or slightly less) than compiled stubs. Hence, it will be useful for small distributed equipment such as handsets, PDAs, etc.
  • In doing the above, improvements to the IIOP protocol engine in terms of size/performance will be made.

Event Channel

Current status:

  • The current Event Channel is running under Orbix, fully distributed, the main test I've running has three processes, one runs the EC itself, one a Supplier and yet another a Consumer.
  • The configuration runs can be done even on the distributed scenario,using a IDL "Scheduling Service".
  • At run-time (no config runs) there is no need to use the Scheduling Service, a faster, collocated implementation for the service is available. Obviously the information is generated on the config runs and linked into the program.

Future work:

  • Port to TAO (on availability of stable IDL compiler).
  • Change to use standard naming service rather than current Orbix _bind/_marker calls.
  • The Naming Service must be ported to TAO in order for suppliers and consumers to obtain an object reference to the Event Channel in a portable and reliable fashion.
  • The Scheduling Service is completely independant of the EC and the configuration of suppliers or consumers. I think it would be convenient to have a library and binaries for it, including a tool to dump the results in a C++ compilable form.

Test & Example Applications

Current status:

The TAO test application is meant to serve as a starting point for real-time tests on the TAO system. It comprises the following parts:

  • Server. The servant component creates multiple CORBA objects (servants), each with different real-time priorities. This priority is implemented by using real-time thread support provided by the operating system. Thus, requests sent to a high-priority servant are handled by a high-priority real-time thread, and those sent to a lower priority servant are handled by correspondingly lower priority threads.
  • Client. The client component binds to the servant objects, and sends a stream of CORBA requests to the servants. It measures the response time, i.e. the time taken for the request to complete successfully. In particular, it measures the time taken for requests sent to the high priority servant to complete. The volume of lower priority requests is configurable. The client is thus able to measure the performance of the high-priority servant in the presence of competition from several lower-priority servants.
  • Clearly, if the ORB endsystem handles the priorities of the various requests correctly, increasing the volume of lower priority requests should not affect the performance seen by the higher priority requests. The application thus serves as a tool to measure and confirm this behavior.

Future work:

  • Study the impact of scheduling & concurrency strategies on performance.
  • Evolve into a testbed for discovering sources of performance non-determinism & priority inversion.

Portable Object Adapter

Current Status:

  • The POA is not integrated into this release.

Future work:

  • Mostly compliant with the ORB Portability Specifications [orbos/97-05-15] published by OMG.
  • Provide extensions of the specification to ensure real-time delivery of messages.

CORBA Services

NAMING SERVICE

The Naming Service is a nonpersistent implementation of the OMG Naming Service specification.

Current status:

  • This implementation works on ORBIX 2.0.

Future work:

  • Port to TAO (on availability of stable IDL compiler).

TRADING SERVICE

The Trading Service is an implementation conforming to OMG Trading Service specification.

Current status:

  • This is a simple trader (registers/withdraws/describes offers and does constraint based lookup for those offers) working on VisiBroker 2.0 v1.51.

Future work:

  • Port to TAO.
  • Implement Type Repository and Admin Interfaces for Trading Service.
