AMH Interface and Design Synopsis.

 

 

C++ Mapping:

 

Consider canonical example of IDL:

 

Interface Stock

{

      exception Invalid_Stock_Symbol {};

module Quoter

      {

            long get_quote( in string stock_name )

raises (Invalid_Stock_Symbol);

      };

};

 

 

This IDL is then compiled using a special flag in the IDL compiler. The proposed one is –GH. The IDL compiler then generates special AMH enabled skeletons. The implementation header file would then look like this:

 

// .h file :

class My_AMH_Quoter : public POA_Stock::AMH_Quoter

{

 public:

       // ORB invokes this method, apps

            // implement Object behavior here

       virtual void get_quote (

           // ... the <RH> argument is used to send response.

// It can be stored for later use.

           AMH_QuoterResponseHandler_ptr rh,

            const char *stock_name);

};

 

Notes:

-        Normal skeletons that are POA_* are also generated by the IDL (?)

-        The memory management rules for the ResponseHandler (RH) are yet to be fully concretized but right now it is assumed that the application will be responsible for deallocating the RH after it is done with it.

-        Also, the RH has special semantics. It can be ‘invoked’ only once. Once it has been invoked, invoking it again will raise a ‘suitable’ exception on the server side (Operation_Not_Possible?).

 

The AMH_QuoterResponseHandler class is generated by the IDL compiler:

 

// This class is implemented by the ORB

class AMH_QuoterResponseHandler

{

 public:

 // Servers use this method to send their responses back to clients

 void get_quote (CORBA::Long return_value);

 

};

 

Exceptions:

 

Follow a similar path as in AMI: create an implied IDL skeleton such as:

namespace POA_Stock

{

      class AMH_QuoterResponseHandler

// or just AMH_QuoterHandler ?

            : public POA_Messaging::ResponseHandler

                                                                           // ------------------------- OK?

      {

        public:

            virtual void get_quote (CORBA::Long l) = 0;

            virtual void get_quote_excep (

                  Stock::AMH_QuoterExceptionHolder *excep) = 0

      };

}

The benefits of using a valuetype to hold an exception are:

-        It is similar to the existing AMI stuff

-        It works with the standard mapping as well as with the "alternative
mapping for dialects without native exceptions" and

-        It is illegal to pass exceptions as arguments in IDL, using a valuetype
might keep the language lawyers at bay.

Responsibilities of ORB/Skeleton/Servant :

 

Mayur:

Here is how AMH is planned to be used. When the request arrives for an AMH enabled servant, the ORB creates a ResponseHandler object (RH). RH now contains enough state in it so that the appropriate reply can be sent back to the client. This RH along with the 'IN' parameters are passed in to the AMH enabled servant. The servant then, depending upon certain criteria, either handles the request immediately or puts it into a 'queue' along with the parameters and returns control back to the ORB. Later at some point, some method in the app looks at the queue and decides to handle some RH in it.

 

How and who marshalls (any of the) out parameters?

    The Response Handler.

 

Who sends the response back to the client?

    The Response Handler and the ORB collaborate for that, as soon as you invoke the callback method on it, well, maybe it gets queued in the I/O subsystem and stuff. The ORB does the work, behind the scenes, using the response handler to hide all the details.

 

Is there any ORB-specific state in the RH?

    Yes, the request ID, the connection used to receive the request, and possibly, some state related to the interceptors.

 

I can guess that there should be, but if the app needs to manipulate it, what interfaces do we provide for the app to safely modify it?

    The application never modifies that state, it only invokes the callback methods to (implicitly) send the response, that may change the state.  For example, a RH should be used only once (there is one response per request), so there probably would be a "fired" state, once set any attempt to invoke the callback methods again should raise an exception.  As far as I can see today,  the app should not need nor have access to any of the RH state, the RH is an opaque handle provided to send back responses.

  The RH should be able to give the same information as a servant in an upcall would have had.
